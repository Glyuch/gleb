# Перестройка ФондыКвест под накопленный портфель (DCA) — дизайн-спека

Дата: 2026-06-20
Статус: согласовано (дизайн); ревью пройдено (4 независимых проверяющих), правки внесены; готовится план реализации
Проект: gleb.finance (Laravel 13 + Fortify + Inertia React; игра — vanilla JS в Blade)
Где живёт игра: `/game` (за авторизацией) → `resources/views/game.blade.php`; сервер `app/Http/Controllers/GameController.php`; контент в БД `game_contents` (активная JSON-версия); модель `app/Models/GameResult.php`.

## 1. Проблема и цель

Сейчас портфель — одно число `you`, и каждый ход весь портфель перекладывается в выбранный инструмент (`you = you * (1 + ret[choice])`). Состава по инструментам в модели нет: итог всегда 100% в инструменте последнего хода — доли показать неоткуда.

Цель: перестроить ядро так, чтобы формировался **реальный диверсифицированный портфель** из 5 инструментов, и в финале (и по ходу) показывался его **состав по долям**.

## 2. Принятые решения

| Вопрос | Решение |
|---|---|
| Механика | **A · DCA (взносы)** — каждый квартал распределяется только новый взнос |
| Стартовый капитал | нет; портфель строится из **12 равных взносов** |
| Размер взноса | **30 000 ₽/квартал** (≈10k/мес), итого 360 000 ₽; `config('game.contribution')` |
| Уже вложенное | не трогается, растёт само по своей доходности |
| Тайминг взноса | взнос НЕ участвует в доходности своего квартала, растёт со **следующего** |
| «Смешанный» | отдельный 5-й инструмент `mix` со своей доходностью `ret[q].mix` (не вычисляется на лету) |
| Инструменты | 5: `bank`, `cash`, `bond`, `stock`, `mix` |
| Состав в финале | **полоса (stacked bar)** на 5 долей + легенда |
| Состав по ходу | **да** — тонкая живая полоса в HUD + рекап «портфель за прошлый квартал ±%» |
| Карточка выбора | показывает **прошлую динамику** инструмента (бейдж «за прошлый квартал ±%» + мини-график) |
| Эталон «Максимум» | **идеал по каждому взносу** (perfect foresight), тайминг `q+1` |
| Эталон «Вклад» | те же взносы, все в `bank`, тот же тайминг |
| Старые данные | **полностью затереть** (`game_results` + `game_events`) |
| Лидерборд | с чистого листа; ранжирование по `score_you` (₽) |
| Источник истины | **сервер** пересчитывает score/benchmarks/composition детерминированно из `choices`; клиентский `score_you` — только сверка |

## 3. Модель движка

### 3.1 Состояние

```
bal = { bank: 0, cash: 0, bond: 0, stock: 0, mix: 0 }
```

Константы: `C = config('game.contribution') = 30000`, `N = 12`, `ret[q][i]` — квартальная доходность инструмента `i ∈ {bank,cash,bond,stock,mix}` в квартале `q` (из активного контента).

### 3.2 Канонический цикл квартала (единый для всех реализаций)

Для `q = 0..11`:

1. **Доходность** на уже вложенное: для каждого `i`: `bal[i] *= (1 + ret[q][i])`.
   В `q=0` балансы нулевые — шаг **холостой** (выполняется как no-op; не считать 0/0 для рекапа).
2. **Рекап + карточки**:
   - Рекап портфеля: `q=0` → показать «старт» (портфеля ещё нет); `q≥1` → `Δ = (sum(bal) − sumBefore)/sumBefore` за шаг 1 (первое осмысленное число появляется на `q=1`).
   - Карточки: на каждой — доходность **за прошлый квартал** `ret[q][i]` (бейдж) + мини-график истории `ret[0..q][i]`.
3. **Выбор**: игрок кладёт взнос в фонд `k`: `bal[k] += C`. Микрокопи-подтверждение: «Взнос 30 000 ₽ вложен в {фонд}. Начнёт работать со следующего квартала.»
4. **Переход**: кнопка «Следующий квартал →» → `q++` → к шагу 1 (см. §9.3 — это единственный триггер перехода после удаления модала).

Итог: `you = bal.bank + bal.cash + bal.bond + bal.stock + bal.mix`.

Так как взнос добавляется ПОСЛЕ доходности квартала `q`, он впервые растёт на `ret[q+1]`. Взнос квартала `q` в инструмент `k` в финале стоит:

```
C * Π_{t=q+1..11} (1 + ret[t][k])
```

Взнос `q=11` добавляется после финальной доходности и не растёт (пустое произведение = 1) — реалистично.

### 3.3 Единый темпоральный фрейм («за прошлый квартал»)

`ret[q]`, реализованная на шаге 1 квартала `q`, — это рыночное движение **только что закрывшегося квартала**, по которому игрок и принимает решение (имитация «выбираем по прошлой доходности»). Его взнос отрабатывает уже на будущей `ret[q+1]`.

- **Единая подпись во всех местах**: «за прошлый квартал» — и на бейдже карточки, и в рекапе HUD, и в прозе. (Нигде не «за этот квартал».)
- **`q=0`**: рекап портфеля = «старт» (холдингов нет), но карточки всё равно показывают `ret[0]` как «за прошлый квартал» — это рыночное движение перед стартом игры (существует независимо от холдингов игрока). Мини-график при единственной точке `ret[0]` → рисуем точкой или скрываем (не ломаную).
- Особенность вклада («фиксирует доходность вперёд») в игре игнорируем — все инструменты, включая `bank`, показываем единообразно по их `ret`. Скорректировать интро-копи, где это противоречит (см. §8.3).

## 4. Эталоны и очки — каноническая опорная формула

Все три величины считаются **по одному алгоритму** и совпадают до рубля между: клиентским движком, клиентским `computeBenchmarks` (бывш. `computeMax`) и серверным `benchmarks()`. Все стартуют от 0, итерируют взносы `q=0..11`, перемножают путь `t=q+1..11` в порядке возрастания `t`, читают `mix` из `ret[q].mix` (никаких `(bond+stock)/2` на лету).

```
C = config('game.contribution'); INSTR = [bank, cash, bond, stock, mix]
score_bank = Σ_{q=0..11} C * Π_{t=q+1..11}(1 + ret[t].bank)
score_max  = Σ_{q=0..11} C * max_{i∈INSTR} Π_{t=q+1..11}(1 + ret[t][i])
you        = результат прогона §3.2 по фактическим choices (сумма bal)
score_you  = round(you)          // round от серверного you
ratio      = max>0 ? round(min(you/max, 1) * 100) : 0   // you и max — серверные
```

Инварианты (defensive-гварды, но всегда верны): `you ≤ score_max`; `score_max ≥ C` (взнос `q=11` пустое произведение = 1). Поэтому `min(...,1)` и `max>0` срабатывают только на float-шум.

«Максимум» в финале не раскрывается (как сейчас).

## 5. Состав портфеля

`composition` — финальные доли/суммы по 5 инструментам:

```
composition = { bank:{amount,share}, cash:{...}, bond:{...}, stock:{...}, mix:{...} }
share[i] = you>0 ? bal[i]/you : 0
```

Сохраняется в `game_results.composition` (json). Цвета: bank=серый (g600), cash=зелёный, bond=синий, stock=красный, mix=амбер (как в интро).

**Правила отрисовки полосы (acceptance, мобильный ~452px после паддинга app):**
- Подписи (₽ + доля%) живут в **легенде**, не внутри тонких сегментов.
- Сегмент с долей ниже порога (~6–8%) → тонкая «нитка» без внутренней подписи.
- Инструмент с нулевой долей → в полосе отсутствует, в легенде может показываться как 0%.
- Легенда из 5 пунктов — 2 колонки / перенос, чтобы не переполнять 480px.

## 7. Сервер: `GameController`

### 7.1 `store()` — сервер как источник истины

- **`choices` становится обязательным и строгим**: `required|array|size:12`; каждый элемент `{ quarter: int 1..12, k: string }`; `quarter` уникальны и покрывают 1..12 (проверять полноту/уникальность серверно, не только по-элементно); `k ∈` множество инструментов активного контента (`Rule::in([...])`). Поле `ret` из payload **убирается** (сервер сам берёт доходности из контента).
- **Сервер прогоняет §3.2** по `ret` активного контента, считает `bal`, `you`, `composition`, а также `score_bank`, `score_max`, `ratio` по §4. **Сохраняет только серверные значения.**
- **Клиентский `score_you` — необязательная сверка**: при `|client_score_you − server_you| > 1₽` залогировать аномалию (событие/лог), но **всё равно сохранить серверное значение и НЕ возвращать 422** игроку. Старое правило `score_you => max:($max+1)` **удалить** (оно не применимо к серверному значению и больше не является механизмом целостности).
- **Неполный/битый `choices`** (закрыл игру на середине и т.п.) → `422`, `GameResult` не пишется. (Подтвердить как приемлемое продуктовое поведение.)
- Логирует событие `result`. Ответ: `promo`, `shopUrl`, `leaderboard`, `rank` (как сейчас).

## 15. Риски / на что смотреть

- **Тайминг `q+1` и единый алгоритм §4** должны совпадать в трёх местах (клиентский движок, `computeBenchmarks`, серверный `benchmarks()`) — до рубля; иначе провалится сверка `score_you`. Текущие `computeMax`/`benchmarks` — другая модель (~561k vs ~402k на одних данных), переписать целиком.

- **Контент-копи**: остаточные «300 000 ₽»/«весь портфель»/«что сейчас выгоднее» сломают нарратив — пройтись по всем 12 кварталам, интро, `hintBase` и **захардкоженным blade-строкам**.
- **Удаление модала** убирает единственный триггер `nextYear` — обязательно завести кнопку перехода (§9.3).
- **5 цветов на 480px**: правила §5 (подписи в легенде, тонкие сегменты «ниткой», легенда в 2 колонки).
- **`DELETE` необратим, но транзакционен**: пользователь подтвердил затирание; `TRUNCATE` не использовать (не откатывается).
- **Разброс `score_you` сужается** при DCA — лидерборд ранжирует уже; при желании перейти на `ratio` (см. §7.3).
- **Источник истины**: сохраняем серверные значения, клиентский `score_you` — только сверка; `ratio` и состав считаются от серверного `you`.
